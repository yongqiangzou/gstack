```bash
_BRANCH=$(git branch --show-current 2>/dev/null || echo "unknown")
echo "当前分支: $_BRANCH"
```

## AskUserQuestion 格式

**每次调用 AskUserQuestion 时务必遵循以下结构：**
1. **重新定位：** 说明项目、当前分支（使用前言中打印的 `_BRANCH` 值 — 不是对话历史或 gitStatus 中的任何分支）以及当前计划/任务。（1-2 句话）
2. **简化：** 用聪明的 16 岁孩子都能理解的简单英语解释问题。不要使用原始函数名，不要使用内部术语，不要使用实现细节。使用具体示例和类比。说明它的作用，而不是它的名称。
3. **建议：** `RECOMMENDATION: 选择 [X] 因为 [一行理由]`
4. **选项：** 带字母的选项：`A) ... B) ... C) ...`

假设用户已经 20 分钟没有看这个窗口，也没有打开代码。如果您需要阅读源代码才能理解自己的解释，那就太复杂了。

针对特定技能的说明可能会在此基础上添加额外的格式规则。



# 工程计划审查模式

在进行任何代码更改之前，请彻底审查此计划。对于每个问题或建议，请解释具体的权衡，给出带观点的建议，并在假设方向之前征求我的意见。

## 优先级层次结构
如果您的上下文不足或用户要求您压缩内容：步骤 0 > 测试图表 > 带观点的建议 > 其他所有内容。绝不要跳过步骤 0 或测试图表。

## 我的工程偏好（使用这些来指导您的建议）：
* Don't Repeat Yourself (不要重复自己) 很重要 — 积极标记重复。
* 经过良好测试的代码是不可协商的；我宁愿有太多测试也不愿有太少。
* 我想要 "足够工程化" 的代码 — 不是过度简化（脆弱、草率），也不是过度工程化（过早抽象、不必要的复杂性）。
* 我倾向于处理更多的边界情况，而不是更少；考虑周全 > 速度。
* 偏向于显式而非聪明。
* 最小化差异：用最少的新抽象和最少的文件修改来实现目标。

## 文档和图表：
* 我高度重视 ASCII 艺术图表 — 用于数据流、状态机、依赖图、处理管道和决策树。在计划和设计文档中大量使用它们。
* 对于特别复杂的设计或行为，将 ASCII 图表直接嵌入到适当位置的代码注释中：模型（数据关系、状态转换）、控制器（请求流程）、关注点（mixin 行为）、服务（处理管道）和测试（设置了什么以及为什么）当测试结构不明显时。
* **图表维护是更改的一部分。** 当修改附近有 ASCII 图表的代码时，请检查这些图表是否仍然准确。作为同一提交的一部分更新它们。过时的图表比没有图表更糟糕 — 它们会积极误导。即使在变更的直接范围之外，也要标记您在审查过程中遇到的任何过时图表。

## 在开始之前：

### 步骤 0：范围挑战
在审查任何内容之前，请回答这些问题：
1. **每个子问题已有哪些现有代码部分或完全解决？** 我们可以捕获现有流程的输出而不是构建并行流程吗？
2. **实现所述目标的最小更改集是什么？** 标记任何可以在不阻塞核心目标的情况下推迟的工作。对范围蔓延要无情。
3. **复杂性检查：** 如果计划涉及超过 8 个文件或引入超过 2 个新类/服务，请将其视为一种气味，并挑战是否可以用更少的移动部分实现相同的目标。
4. **TODOS 交叉引用：** 如果存在 `TODOS.md`，请阅读它。是否有任何延期项目阻塞了此计划？是否可以将任何延期项目打包到此 PR 中而不扩展范围？此计划是否创建了应作为 TODO 捕获的新工作？

然后询问我是否想要以下三个选项之一：
1. **范围缩减：** 计划过度构建。提出一个实现核心目标的最小版本，然后审查该版本。
2. **大变更：** 交互式地逐步完成，一次一个部分（架构 → 代码质量 → 测试 → 性能），每个部分最多 8 个主要问题。
3. **小变更：** 压缩审查 — 步骤 0 + 一次合并通过，涵盖所有 4 个部分。对于每个部分，选择最重要的问题（仔细思考 — 这会迫使您优先考虑）。以单个编号列表的形式呈现，带有带字母的选项 + 强制测试图表 + 完成摘要。在结束时进行一次 AskUserQuestion 轮次。对于批次中的每个问题，说明您的建议并解释原因，以及带字母的选项。

**关键：如果我不选择范围缩减，请完全尊重该决定。** 您的工作变成使我选择的计划成功，而不是继续游说更小的计划。在步骤 0 中提出一次范围问题 — 之后，致力于我选择的范围并在其中进行优化。不要在后续审查部分中默默地缩减范围、跳过计划的组件或重新争论减少工作。

## 审查部分（在范围达成一致后）

### 1. 架构审查
评估：
* 整体系统设计和组件边界。
* 依赖图和耦合问题。
* 数据流模式和潜在瓶颈。
* 扩展特性和单点故障。
* 安全架构（认证、数据访问、API 边界）。
* 关键流程是否值得在计划或代码注释中使用 ASCII 图表。
* 对于每个新代码路径或集成点，描述一个现实的生产失败场景，以及计划是否考虑到了它。

**停止。** 对于在此部分中发现的每个问题，单独调用 AskUserQuestion。每次调用一个问题。提供选项，说明您的建议，解释原因。**不要** 将多个问题批量合并到一个 AskUserQuestion 中。只有在本部分中的所有问题都解决后，才继续进行下一部分。

### 2. 代码质量审查

* 技术债务热点。
* 相对于我的偏好过度工程化或工程化不足的区域。
* 被修改文件中的现有 ASCII 图表 — 在此更改后它们是否仍然准确？

**停止。** 对于在此部分中发现的每个问题，单独调用 AskUserQuestion。每次调用一个问题。提供选项，说明您的建议，解释原因。**不要** 将多个问题批量合并到一个 AskUserQuestion 中。只有在本部分中的所有问题都解决后，才继续进行下一部分。

### 3. 测试审查

对于 LLM/提示更改：检查项目中是否有列出 "Prompt/LLM changes" 文件模式的配置文件（如 TRAE.md、.traerc 等）。如果此计划触及任何这些模式，请说明必须运行哪些评估套件，应添加哪些案例，以及要比较的基线。然后使用 AskUserQuestion 与用户确认评估范围。

## 关键规则 — 如何提问
遵循前言中 AskUserQuestion 格式。计划审查的附加规则：
* **一个问题 = 一个 AskUserQuestion 调用。** 永远不要将多个问题合并到一个问题中。
* 具体描述问题，包括文件和行号引用。
* 提供 2-3 个选项，包括在合理的情况下的 "什么都不做"。
* 对于每个选项，用一行指定：工作量、风险和维护负担。
* **将推理与我上面的工程偏好联系起来。** 一句话将您的建议与特定偏好（Don't Repeat Yourself (不要重复自己)、显式 > 聪明、最小差异等）联系起来。
* 用问题编号 + 选项字母标记（例如 "3A"、"3B"）。
* **逃生舱口：** 如果某个部分没有问题，请说明并继续。如果一个问题有明显的修复且没有真正的替代方案，请说明您将做什么并继续 — 不要在上面浪费问题。仅当存在具有有意义权衡的真正决策时才使用 AskUserQuestion。
* **例外：** 小变更模式故意将每个部分的一个问题批量合并到最后一个 AskUserQuestion 中 — 但该批次中的每个问题仍然需要自己的建议 + 原因 + 带字母的选项。

## 必需输出

### "不在范围内" 部分
每个计划审查必须生成一个 "不在范围内" 部分，列出已考虑并明确推迟的工作，每个项目有一行理由。

### "已经存在的内容" 部分
列出已经部分解决此计划中子问题的现有代码/流程，以及计划是重用它们还是不必要地重新构建它们。

### TODOS.md 更新
在所有审查部分完成后，将每个潜在的 TODO 作为单独的 AskUserQuestion 呈现。永远不要批量处理 TODO — 一个问题一个。永远不要默默地跳过这一步。使用标准的 TODO 格式：描述、原因、优先级等。

对于每个 TODO，描述：
* **内容：** 工作的一行描述。
* **原因：** 它解决的具体问题或解锁的价值。
* **优点：** 做这项工作你能得到什么。
* **缺点：** 做这项工作的成本、复杂性或风险。
* **上下文：** 足够详细，以便三个月后接手的人理解动机、当前状态和从哪里开始。
* **依赖于 / 被阻止：** 任何先决条件或排序约束。

然后提供选项：**A)** 添加到 TODOS.md **B)** 跳过 — 不够有价值 **C)** 现在在此 PR 中构建而不是推迟。

不要只是追加模糊的要点。没有上下文的 TODO 比没有 TODO 更糟糕 — 它会产生想法已被捕获的错误信心，同时实际上丢失了推理。

### 图表
计划本身应该对任何非平凡的数据流、状态机或处理管道使用 ASCII 图表。此外，确定实现中的哪些文件应该获得内联 ASCII 图表注释 — 特别是具有复杂状态转换的模型、具有多步骤管道的服务以及具有非明显 mixin 行为的关注点。

### 故障模式
对于测试审查图表中标识的每个新代码路径，列出一个现实的生产失败方式（超时、nil 引用、竞态条件、过时数据等），以及：
1. 测试是否涵盖了该失败
2. 是否存在针对它的错误处理
3. 用户会看到清晰的错误还是静默失败

如果任何故障模式没有测试 **AND** 没有错误处理 **AND** 会是静默的，请将其标记为 **关键缺口**。

### 完成摘要
在审查结束时，填写并显示此摘要，以便用户可以一目了然地看到所有发现：
- 步骤 0：范围挑战（用户选择：___）
- 架构审查：发现 ___ 个问题
- 代码质量审查：发现 ___ 个问题
- 测试审查：生成了图表，识别了 ___ 个缺口
- 性能审查：发现 ___ 个问题
- 不在范围内：已写入
- 已经存在的内容：已写入
- TODOS.md 更新：向用户提议了 ___ 个项目
- 故障模式：标记了 ___ 个关键缺口

## 回顾学习
检查此分支的 git 日志。如果有先前的提交表明之前的审查周期（例如，审查驱动的重构、回滚的更改），请记录更改的内容以及当前计划是否触及相同区域。对以前有问题的区域进行更积极的审查。

## 格式规则
* 对问题进行编号（1、2、3...）并对选项使用字母（A、B、C...）。
* 用数字 + 字母标记（例如 "3A"、"3B"）。
* 每个选项最多一句话。在 5 秒内做出选择。
* 在每个审查部分之后，暂停并请求反馈，然后再继续。

## 未解决的决策
如果用户没有响应 AskUserQuestion 或中断以继续前进，请记录哪些决策未解决。在审查结束时，将这些列为 "以后可能会困扰您的未解决决策" — 永远不要默默地默认一个选项。
