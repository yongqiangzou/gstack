# 预上线审查清单

## 说明

审查 `git diff origin/main` 输出，查找以下列出的问题。要具体 — 引用 `file:line` 并建议修复。跳过一切正常的内容。只标记真正的问题。

**两次审查：**
- **第一次（关键）：** 首先运行SQL与数据安全和信任边界审查。这些可能会阻止 `/ship`。
- **第二次（信息性）：** 运行所有剩余类别。这些包含在PR正文中，但不会阻止。

**输出格式：**

```
预上线审查：N个问题（X个关键，Y个信息性）

**关键**（阻止/ship）：
- [file:line] 问题描述
  修复：建议的修复方案

**问题**（非阻止）：
- [file:line] 问题描述
  修复：建议的修复方案
```

如果没有发现问题：`预上线审查：未发现问题。`

保持简洁。每个问题：一行描述问题，一行修复方案。没有前言，没有总结，没有"总体看起来不错"。

---

## 审查类别

### 第一次 — 关键

#### SQL与数据安全
- SQL中的字符串插值（即使值使用`.to_i`/`.to_f` — 也要使用`sanitize_sql_array`或Arel）
- TOCTOU竞态条件：应该使用原子`WHERE` + `update_all`的检查-然后-设置模式
- `update_column`/`update_columns`绕过了有或应该有约束的字段的验证
- N+1查询：循环/视图中使用的关联缺少`.includes()`（特别是头像、附件）

#### 竞态条件与并发
- 没有唯一性约束或`rescue RecordNotUnique; retry`的读-检查-写（例如，`where(hash:).first`然后`save!`而不处理并发插入）
- 在没有唯一数据库索引的列上使用`find_or_create_by` — 并发调用可能会创建重复项
- 不使用原子`WHERE old_status = ? UPDATE SET new_status`的状态转换 — 并发更新可能会跳过或重复应用转换
- 在用户控制的数据上使用`html_safe`（XSS）— 检查任何`.html_safe`、`raw()`或插值到`html_safe`输出的字符串

#### 信任边界
- 写入数据库或传递给邮件程序的生成值（电子邮件、URL、名称）没有格式验证。在持久化之前添加轻量级保护（`EMAIL_REGEXP`、`URI.parse`、`.strip`）。
- 接受结构化工具输出（数组、哈希）而不在数据库写入前进行类型/形状检查。

#### 枚举与值完整性
当差异引入新的枚举值、状态字符串、层级名称或类型常量时：
- **跟踪它通过每个消费者。** 阅读（不要只是grep — 要阅读）每个切换、过滤或显示该值的文件。如果任何消费者不处理新值，标记它。常见遗漏：向前端下拉菜单添加值，但后端模型/计算方法不持久化它。
- **检查允许列表/过滤数组。** 搜索包含兄弟值的数组或`%w[]`列表（例如，如果向层级添加"revise"，则查找每个`%w[quick lfg mega]`并验证"revise"是否在需要的地方包含）。
- **检查`case`/`if-elsif`链。** 如果现有代码在枚举上分支，新值是否会落入错误的默认分支？
要做到这一点：使用Grep查找所有引用兄弟值的地方（例如，grep "lfg"或"mega"以找到所有层级消费者）。阅读每个匹配项。此步骤需要读取差异外的代码。

### 第二次 — 信息性

#### 条件副作用
- 分支条件但忘记在一个分支上应用副作用的代码路径。例如：项目被提升为已验证，但URL仅在次要条件为真时附加 — 另一个分支在没有URL的情况下提升，创建不一致的记录。
- 声称发生了某个动作但该动作被条件性跳过的日志消息。日志应该反映实际发生的情况。

#### 魔术数字与字符串耦合
- 在多个文件中使用的裸数字字面量 — 应该是一起记录的命名常量
- 其他地方用作查询过滤器的错误消息字符串（grep该字符串 — 是否有任何东西匹配它？）

#### 死代码与一致性
- 赋值但从未读取的变量
- PR标题与VERSION/CHANGELOG文件之间的版本不匹配
- 不准确描述更改的CHANGELOG条目（例如，"从X更改为Y"，但X从未存在）
- 代码更改后描述旧行为的注释/文档字符串

#### 提示问题
- 提示中的0索引列表（大模型可靠地返回1索引）
- 提示文本列出的可用工具/功能与`tool_classes`/`tools`数组中实际连接的不匹配
- 在多个地方声明的可能会漂移的单词/令牌限制

#### 测试缺口
- 断言类型/状态但不断言副作用的负面路径测试（URL附加？字段填充？回调触发？）
- 断言字符串内容但不检查格式（例如，断言标题存在但不检查URL格式）
- 缺少`.expects(:something).never`，当代码路径应明确不调用外部服务时
- 没有集成测试验证执行路径端到端工作的安全实施功能（阻止、速率限制、认证）

#### 加密与熵
- 截断数据而不是哈希（最后N个字符而不是SHA-256）— 熵更少，更容易碰撞
- 用于安全敏感值的`rand()` / `Random.rand` — 改用`SecureRandom`
- 对秘密或令牌的非恒定时间比较（`==`）— 容易受到定时攻击

#### 时间窗口安全
- 假设"今天"覆盖24小时的日期键查找 — 在PT早上8点报告，只看到当天键下的午夜→早上8点
- 相关功能之间不匹配的时间窗口 — 一个使用小时桶，另一个对相同数据使用日键

#### 边界处的类型强制转换
- 跨越Ruby→JSON→JS边界的值，类型可能会改变（数字vs字符串）— 哈希/摘要输入必须规范化类型
- 序列化前不调用`.to_s`或等效方法的哈希/摘要输入 — `{ cores: 8 }`与`{ cores: "8" }`产生不同的哈希

#### 视图/前端
- 部分中的内联`<style>`块（每次渲染都重新解析）
- 视图中的O(n*m)查找（循环中使用`Array#find`而不是`index_by`哈希）
